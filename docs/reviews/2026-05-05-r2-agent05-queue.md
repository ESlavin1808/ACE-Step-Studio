## R1 — NEW — LOW — `setPendingClickCount(0)` clamps mid-bulk pending undercount

**Refs:** `App.tsx:832`

Walk-through requested by reviewer:
1. User clicks Bulk=10. `incrementPendingClicks(10)` → `pendingClickCount = 10`.
2. 5 of the 10 LLM pre-flight chain steps fire `onGenerate` → 5 `beginPollingJob` register → 5× `decrementPendingClicks(1)`. Counter: pending=5, active=5.
3. User clicks Cancel All. `cancelAllGenerations`: `activeJobsRef.clear()`, `setActiveJobCount(0)`, `drainQueueWaiters()`, `setPendingClickCount(0)`. Counter: pending=0, active=0.
4. The other 5 chain steps were parked on `await waitForJobsToDrain()`. `drainQueueWaiters` resolves them → each one runs OR `client.generate(...)` → resolves a `perClickDraft` → for-loop in CreatePanel was already executed (loop is OUTSIDE the chain step, after `await llmPreflightQueueRef.current` at `:1571`). Wait — re-read.

Re-checking CreatePanel structure: the chain step itself only does the OR call. The for-loop at `:1612` runs AFTER `await llmPreflightQueueRef.current` resolves. So during cancel-all the awaiting `handleGenerate` invocations (one per click) are blocked at `:1571` `perClickDraft = await llmPreflightQueueRef.current`. Good — they unblock when their chain step settles.

But: the chain step's `try { await waitForJobsToDrain(); }` resolves on `drainQueueWaiters()`, then the chain step proceeds to call OR (still spawns the request — there's no AbortController exposed across the chain step boundary). Once OR returns → chain step resolves → CreatePanel continues past `:1571`, runs the for-loop, fires `bulkCount` `onGenerate` calls.

Each of those `onGenerate` calls reaches App.handleGenerate → `beginPollingJob` (success) → `decrementPendingClicks(1)`. But `pendingClickCount` is already 0 (the cancel-all set it). `Math.max(0, 0 - 1) = 0`. Counter stays at 0 — visually correct (no leak). But this means the badge briefly DOES NOT REFLECT the in-flight clicks during the post-cancel restart window. The user clicked cancel-all but 5 new jobs spawn. Visual state lies.

**Severity LOW.** Counter is monotonic and clamped, so no permanent bad state. Worst case is a few seconds of "0 pending, jobs appearing" confusion. Acceptable; flagging only.

**Optional fix:** Inside `cancelAllGenerations`, also reset `llmPreflightQueueRef.current = Promise.resolve(null)` (would require exposing the ref to App or adding a callback). That cuts the parked chain steps off entirely so the cancelled bulk truly stops. The current fix avoids the deadlock but doesn't actually cancel the in-flight LLM calls.

## R2 — NEW — LOW — `drainQueueWaiters` race with concurrent `cleanupJob`

**Refs:** `App.tsx:98-103`, `:744-762`

`drainQueueWaiters` body:
```ts
if (activeJobsRef.current.size !== 0) return;
const waiters = queueDrainResolversRef.current;
queueDrainResolversRef.current = [];
waiters.forEach((r) => r());
```

If two cleanups happen back-to-back in the same React batch (impossible in JS single-threaded event loop, so a non-issue), the second call sees `[]` and does nothing. JS is single-threaded so the swap-then-iterate is atomic. Safe.

In `cancelAllGenerations` ordering: `activeJobsRef.current.clear()` runs at `:823` BEFORE `drainQueueWaiters()` at `:830`. So at drain time, `activeJobsRef.size === 0` → guard passes → waiters fire. Correct.

If `drainQueueWaiters` were ever called BEFORE `activeJobsRef.clear()`, the guard would early-return and waiters would orphan. Order-of-operations is currently correct in all four call sites (`cleanupJob`, `cancelAllGenerations`). Confirming: not a bug.

## R3 — NEW — LOW — Chain firewall + drain hang scenario

**Refs:** `CreatePanel.tsx:1543-1569`, `App.tsx:765-783` (`cancelGeneration` for single job)

Reviewer's scenario: pre-flight running, user clicks single-job cancel. `cancelGeneration` (App.tsx:765) does NOT call `drainQueueWaiters()`. But it also doesn't `clear()` `activeJobsRef` — it just `delete`s nothing (`activeJobsRef.current.get(jobId)` → may return a job; the song state is updated to `cancelled`, but the job IS NOT REMOVED from the ref). Re-reading:

```ts
const jobData = activeJobsRef.current.get(jobId);
if (jobData) {
  clearInterval(jobData.pollInterval);
  setSongs(prev => prev.map(...));   // update stage to 'cancelled'
}
```

The job entry stays in `activeJobsRef.current` (no `delete`). So `activeJobsRef.size` is unchanged. `waitForJobsToDrain` still waits for it. BUT polling is stopped — `cleanupJob` will never fire for this job, because the only path to `cleanupJob` is via the polling interval. The cancelled job is now a permanent occupant of `activeJobsRef`.

**Consequence:** If the user single-cancels a job and then clicks Создать, the next click's `await waitForJobsToDrain()` blocks on a job that will never drain. Click hangs at the drain barrier indefinitely.

**Severity:** This is a **HIGH** in disguise. The "fix" for H1 (`cancelAllGenerations` adds drain) didn't propagate to single-job cancel. Single-cancel was already buggy in R1 but R1 didn't call this out separately because the cancelled card is still visible — the user might assume "oh, I cancelled one, others still queue". Verifying:

- `cancelGeneration` at line 765-783 sets stage='cancelled' but leaves entry in `activeJobsRef`.
- Next bulk click chain step `await waitForJobsToDrain()` → waits forever.
- User must click Reset (`resetSingleJob`) to actually `delete` from the ref. `resetSingleJob` (`:786-806`) DOES delete and `setActiveJobCount`, but does NOT call `drainQueueWaiters()`. So even after Reset, parked chain steps stay parked.

**This is a regression-class issue.** Promoting from "noted" to its own finding:

## R4 — NEW — HIGH — `cancelGeneration` and `resetSingleJob` still don't drain queue waiters → single-cancel deadlock

**Refs:** `App.tsx:765-783`, `:786-806`

The fix in `cancelAllGenerations` (H1) was correct but partial. The other two cancel-class entry points still leak parked waiters:

- **`cancelGeneration`** (single soft-cancel): does not delete from `activeJobsRef`. Polling stopped, song marked `cancelled`. The active-jobs count is unchanged. Any chain step parked on `waitForJobsToDrain()` waits for this never-drainable job forever. Next click → hang.
- **`resetSingleJob`** (single hard-cancel + GPU reset): DOES delete from `activeJobsRef` and updates `activeJobCount`, but does NOT call `drainQueueWaiters()`. If size goes to 0 from a non-zero state, parked waiters STAY parked. Next click → hang.

The prior review's H1 fix sketch explicitly said "`resetSingleJob` should call `drainQueueWaiters()` after the `delete()` on parity with `cleanupJob`." That guidance was not applied.

**Repro for `resetSingleJob`:**
1. Click Создать → click 1 active.
2. Click Создать again → click 2 chain step parks at `waitForJobsToDrain()`.
3. Click Reset on click 1's card. `resetSingleJob` runs, `activeJobsRef` becomes empty.
4. Click 2's chain step is STILL parked. `drainQueueWaiters` was never called.
5. Click 2 hangs forever — same deadlock as the original H1.

**Repro for `cancelGeneration`:**
1. Click Создать → click 1 active.
2. Click cancel button (soft) on click 1's card. Polling stops, `activeJobsRef` still contains the entry.
3. Click Создать again → click 2 parks at `waitForJobsToDrain()` because `activeJobsRef.size === 1` (the cancelled-but-not-deleted job).
4. The cancelled job will never drain (no polling to call `cleanupJob`).
5. Click 2 hangs forever.

**Fix sketch:**
```diff
@@ App.tsx:782 cancelGeneration
   if (jobData) {
     clearInterval(jobData.pollInterval);
+    activeJobsRef.current.delete(jobId);
+    setActiveJobCount(activeJobsRef.current.size);
+    if (activeJobsRef.current.size === 0) {
+      setIsGenerating(false);
+      drainQueueWaiters();
+    }
     setSongs(prev => prev.map(s =>
       s.id === jobData.tempId ? { ...s, isGenerating: false, stage: 'cancelled' } : s
     ));
   }

@@ App.tsx:805 resetSingleJob
   if (jobData) {
     activeJobsRef.current.delete(jobId);
     setSongs(prev => prev.filter(s => s.id !== jobData.tempId));
     setActiveJobCount(activeJobsRef.current.size);
     if (activeJobsRef.current.size === 0) {
       setIsGenerating(false);
+      drainQueueWaiters();
     }
   }
```

Plus add `drainQueueWaiters` to dep arrays.

## R5 — NEW — LOW — TS chain typing holds

**Refs:** `CreatePanel.tsx:268, 1543`

`llmPreflightQueueRef` is declared as `useRef<Promise<SongDraft | null>>(Promise.resolve(null))`. The new chain:
```ts
llmPreflightQueueRef.current.catch(() => null).then(async () => {...})
```

`.catch(() => null)` on `Promise<SongDraft | null>` returns `Promise<SongDraft | null>` (the `null` from catch widens, but `null` is already in the union — no widening). `.then(async fn)` returns `Promise<ReturnType<fn>>` where the async fn returns `SongDraft | null` (`return await client.generate(...)` → `SongDraft`, `return null` in catch). So the assignment back to `llmPreflightQueueRef.current` is type-compatible. TS compiles. ✅

## R6 — NEW — LOW — In-flight LLM cannot be aborted from outside chain step

**Refs:** `CreatePanel.tsx:1553`

`const ac = new AbortController();` is created INSIDE the chain step, used only by the local `client.generate(... , { signal: ac.signal })`.
