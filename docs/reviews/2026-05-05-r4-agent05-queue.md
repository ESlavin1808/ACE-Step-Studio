## Mental simulation — same scenarios as R3, re-walked at HEAD

Re-run to confirm batch 5's type-only widening did not silently change runtime semantics.

### Scenario A — bulk=10 stacked clicks
- `pendingClickCount` increments to 10 across rapid clicks.
- 10 chain steps queued on `llmPreflightQueueRef.current` (each with `.catch(() => null).then(...)`).
- Each step awaits `waitForJobsToDrain()`, then runs `client.generate(...)` with a *fresh* `AbortController`.
- Result on success: caller awaits `perClickDraft`, runs the for-loop in `CreatePanel`, fires `bulkCount` `onGenerate` calls per click → `App.handleGenerate` → POST `/api/generate` → `beginPollingJob` registers in `activeJobsRef` → `decrementPendingClicks(1)`.
- **Identical to R3.** No type-cast change in `handleGenerate` alters this control flow because the values read (`params.repaintMode`, `params.dcwEnabled`, `params._tempId`, ...) come from the same `params` reference and are still passed through to `generateApi.startGeneration(...)`.

### Scenario B — cancel-all mid-flight
- Step #1's `client.generate` is in flight. User clicks Cancel All.
- `cancelAllGenerations` POSTs `/api/generate/cancel-all`, walks `activeJobsRef` (empty here), `setActiveJobCount(0)`, `setIsGenerating(false)`, `drainQueueWaiters()`, `setPendingClickCount(0)`.
- Step #1's `AbortController` is unreachable from outside its closure → its OR call completes naturally → for-loop fires → cards spawn.
- **Identical to R3-NEW-1.** Visual contract violation persists; no state corruption.

### Scenario C — cancel-single (R4 fix verification)
- 3 jobs polling, user clicks the X on song A.
- `cancelGeneration(jobAId)` → POST `/api/generate/cancel/{id}` → `clearInterval` → `activeJobsRef.delete` → `setActiveJobCount(2)` → `drainQueueWaiters()` (returns early, size=2) → song row marked `stage='cancelled'`.
- Songs B, C continue polling normally. Next bulk click's pre-flight no longer hangs.
- **Identical to R3.** Still working.

### Type-widening risk check
The R4 diff in `handleGenerate` reads, e.g., `params.dcwEnabled` instead of `(params as any).dcwEnabled`. Both expressions resolve to the same property at runtime — TypeScript erases types. The only observable change would be if some field was previously **typed differently** in `GenerationParams` than the way the raw JS object actually held it (e.g. typed as `string` but the runtime value was `{...}` and the cast hid the mismatch). Spot-check of the new `app/types.ts` additions vs the body in `handleGenerate`:

- `_tempId?: string` ↔ used as `const preCreatedId = params._tempId;` then compared to `string | undefined`. ✓
- `pollinations?: { ... }` ↔ passed straight through to the POST body. ✓
- `dcwEnabled?: boolean`, `dcwScaler?: number`, etc. — all whitelisted into the body unchanged. ✓
- `loraLoaded?: boolean` — pass-through. ✓
- `prompt?: string` — pass-through (this also fixed an upstream silent drop on the server route per agent07 R3 L6, but that's outside this agent's scope).

**No semantic drift.** Type-only change.
