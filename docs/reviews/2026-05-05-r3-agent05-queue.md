## R3-NEW-1 — LOW — Cancel-all still spawns up to N jobs after the user clicks Cancel-All (R2 R6 / L1 reaffirmed)

**Refs:** `CreatePanel.tsx:1543-1569`, `App.tsx:818-843`

Mental simulation requested:

1. User clicks **Создать** 5 times rapidly with bulk=1. `pendingClickCount = 5`. Each click calls `incrementPendingClicks(1)` and chains a step on `llmPreflightQueueRef.current`.
2. Chain step #1 starts: `await waitForJobsToDrain()` resolves immediately (no active jobs yet), then it allocates `new OpenRouterProvider()`, `new AbortController()`, calls `client.generate(..., { signal: ac.signal })`. The `ac` reference does not escape the closure.
3. While step #1's `client.generate` is in flight (e.g. 8 s), user clicks **Cancel All**.
4. `cancelAllGenerations` runs:
   - Hits `/api/generate/cancel-all`.
   - Iterates `activeJobsRef` (empty here — nothing has reached `beginPollingJob` yet).
   - `setActiveJobCount(0)`, `setIsGenerating(false)`, `drainQueueWaiters()`, `setPendingClickCount(0)`.
5. **Step #1's `client.generate` is NOT aborted.** No external code has a handle on its `AbortController`. The HTTP request to OpenRouter completes naturally, returns a `SongDraft`. Step #1 resolves.
6. Caller of step #1 (the `await llmPreflightQueueRef.current` at `:1571`) wakes, `perClickDraft` is non-null, the for-loop runs at `:1612` and fires `bulkCount` `onGenerate` calls (= 1 here).
7. `App.handleGenerate` runs: `setIsGenerating(true)`, POST `/api/generate`, eventually `beginPollingJob` registers the job in `activeJobsRef`, then `decrementPendingClicks(1)` → `Math.max(0, 0-1) = 0` (counter already at 0 from step 4).
8. Steps #2-5 inherit the chained promise. After step #1 resolves, step #2's `.then(async () => {...})` runs: `waitForJobsToDrain()` returns immediately if step #1's job hasn't yet been registered, OR waits until that job drains. Either way, step #2 also fires its OR call and follows the same path → another job spawns.

**Net effect:** all 5 clicks spawn jobs *after* the user clicked Cancel-All. The visual contract "I clicked Cancel All, nothing should happen" is broken. `pendingClickCount` clamps at 0 so the badge doesn't lie about counts, but new cards appear in the song list with `stage='stageStartingTrack'` despite the user explicitly cancelling.

**Severity LOW.** No leak (counter is monotonic-clamped, jobs are real and cancellable individually via R4-fixed `cancelGeneration` / `resetSingleJob`). UX-confusing, not state-corrupting.

**Fix sketch (deferred — same as R2 L1):**
- Promote `AbortController` out of the chain step closure: hold a `Set<AbortController>` on a ref in CreatePanel; `cancelAllGenerations` (called from App) needs a callback prop to abort all in-flight ACs and reset `llmPreflightQueueRef.current = Promise.resolve(null)`.
- Or: gate the post-`waitForJobsToDrain` body on a "user-cancelled" generation token. When `cancelAllGenerations` fires it bumps the token; chain steps check the token before calling OR and bail with `null` if it changed.

## R3-NEW-3 — LOW — Race between `resetSingleJob` and an in-flight `beginPollingJob` registration

**Refs:** `App.tsx:792-816`, `beginPollingJob` (per prior reviews lives near `:1207`).

Reviewer's edge case: bulk=5, 3 jobs are already in `activeJobsRef` polling, 2 are still completing their POST → `beginPollingJob` round-trip. User clicks Reset on job #1.

`resetSingleJob`:
1. Hits `/api/generate/reset` (interrupts Gradio across the board).
2. `activeJobsRef.current.delete(jobId)` for job #1.
3. `setActiveJobCount(activeJobsRef.current.size)` → 2.
4. `setIsGenerating(false)` only if size === 0 — here size === 2, so left as-is.
5. `drainQueueWaiters()` — guard `if (activeJobsRef.current.size !== 0) return;` → returns immediately. Waiters not woken (correct: jobs 2, 3 still active).

But the hard `/api/generate/reset` interrupted Gradio for **all** generations, not just job #1. Jobs 2 and 3's polling will see `failed` or `error` on their next tick, eventually call `cleanupJob`, which deletes them from `activeJobsRef` and (per its own implementation) calls `drainQueueWaiters`. So waiters do eventually wake.

Meanwhile click #4's POST is in flight. When it completes and `beginPollingJob` registers job #4, the size goes from 0/1/2 → 1/2/3 (depending on cleanup ordering). If a chain step was parked between the cleanup and the register, it could see size=0 and proceed past `waitForJobsToDrain`, while in parallel the new POST resolves and registers job #4 — meaning two pre-flight chain steps run concurrently for a brief window. They are FIFO-chained so they actually serialise on the promise chain, and each gets its own per-click `perClickDraft` — no shared mutation. Confirmed safe.

**Severity LOW.** Documenting only, not a bug.

## R3-NEW-4 — LOW — `resetSingleJob` calls `/api/generate/reset` (global GPU interrupt) for a *single* card

**Refs:** `App.tsx:797-801`

`resetSingleJob` reaches the **global** Gradio reset endpoint (`/api/generate/reset`), not a per-job interrupt. So clicking Reset on song A while songs B, C are still generating will GPU-interrupt B and C as collateral damage. Their polling will pick up `failed`, then `cleanupJob` will mark them errored.

This is consistent with how Gradio's predict queue works — there's only one worker so any interrupt is global. **Documenting** rather than flagging: the function name is misleading ("singleJob") given the actual blast radius. A user with 3 in-flight songs who clicks Reset on the first will lose all three.

**Severity LOW.** Pre-existing, not a R3 regression.
