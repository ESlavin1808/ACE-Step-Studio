## 3. `openrouterModel` nullability drift · **MIN**

Five touch-points, four different stances:

| Layer | Type | Ref |
|---|---|---|
| `types.ts:Song.openrouterModel` | `string \| null` | `app/types.ts:26` |
| `types.ts:GenerationParams.openrouterModel` | `string` (no `null`, no `undefined` — wait, optional `?` makes it `string \| undefined`) | `app/types.ts:151` |
| `api.ts:GenerationParams.openrouterModel` | `string \| null \| undefined` | `app/services/api.ts:373` |
| `api.ts:Song.openrouterModel` | `string \| undefined` (no `null`) | `app/services/api.ts:115` |
| Backend `GenerateBody.openrouterModel` | `string \| undefined` | `app/server/src/routes/generate.ts:224` |
| DB column | `openrouter_model TEXT` (nullable) | `routes/generate.ts:633,670` writes `aceStatus.result.... openrouter_model || null` |
| App.tsx send site | `(params as any).openrouterModel` — `any` | `app/App.tsx:1134` |

Round-trip: client sends `null` (when user has no OpenRouter model selected). `JSON.stringify({x: null})` → `"x":null`. Backend destructures, hands to ACE-Step. ACE-Step ignores it. Backend writes `openrouter_model || null` to DB → `null`. On read, `transformSongs` does `s.openrouter_model || s.openrouterModel` (`api.ts:133`) — both `null` and `undefined` collapse to `undefined` here because `null || undefined === undefined`. The downstream `Song` type says `string | null` — but the field is actually never `null` after transform, only `undefined`. Drift, but functionally a no-op.

**Suggest:** pick one. `string | null` everywhere (matches DB nullability semantics, what `GET /api/songs` actually returns from the DB before transform), and remove the transform's `|| null` fallback so it doesn't silently flip `null` to `undefined`.

Refs above. Diff sketch:


## 4. `pollinations` defaults double-applied · **MIN**

Frontend `pollinationsStorage.getConfig()` always populates `width/height/seedMode/enhance/nologo/safe` (DEFAULT_POL_CONFIG covers these — `pollinations/types.ts:18-27` makes them all required, so storage shape can never have `undefined` here).

`CreatePanel.tsx:1640-1660` reads `polCfg` and copies each field into the wire payload — they're all defined.

Backend `routes/generate.ts:541-548` then re-applies defaults via `?? 1024`, `?? 'song'`, `?? true`:

```ts
const polCfg: PollinationsCoverConfig = {
  enabled: true,
  apiKey: pol.apiKey || '',
  model: pol.model,
  width: pol.width ?? 1024,
  height: pol.height ?? 1024,
  seedMode: pol.seedMode ?? 'song',
  enhance: pol.enhance ?? true,
  nologo: pol.nologo ?? true,
  safe: pol.safe ?? true,
```

Defensive but redundant. The defaults at the backend disagree with `pollinations/types.ts` defaults (e.g. backend defaults `enhance=true`, but `pollinations/types.ts` doesn't say what the default ought to be — the actual default lives in `pollinations/storage.ts` which I didn't open here). If those two ever drift, you get behaviour "I turned `enhance` off in the UI, retook a song, and it came back enhanced" — the `?? true` would mask `false` only if the value were `undefined`, not `false`, so this particular case is safe. But `apiKey: pol.apiKey || ''` flips empty string to empty string AND null/undefined to empty string — the `||` is laxer than `??` and matches loose intent here. OK.

Recommendation (NIT): drop the backend defaults and require the strict shape. OK in practice but loose by type.

## 5. `_tempId` magic field tunneled via `as any` · **MIN**

`CreatePanel.tsx:1617`, `1731`: writes `_tempId: tempIdForThisJob` into the `onGenerate` payload object. The payload's TS type is `GenerationParams`, which has no `_tempId` field. This compiles only because TS does excess-property checks for object-literals and ALL those object literals are passed to `onGenerate(params: GenerationParams)` — so it would normally error. Why does it compile?

Two object literals (custom-mode `1616-1729` and simple-mode `1730-1761`) are passed via the ternary `(effectiveCustomMode ? {...} : {...})`. TS object-literal excess checks DO apply through a ternary. So this must be compiling for one of these reasons:

1. The fields `_tempId, dcwEnabled, dcwMode, ...` don't trigger excess-property errors because the TYPE of the literal is widened by some other field (e.g. one of the literals contains `as any` or `as const`). Quick scan: `1751: as 'mp3' as const`, `1752: as 'ode' as const` — those are local casts, don't widen. But `1633: randomSeed || i > 0` returns `boolean` — fine. No literal-wide `as any`.
2. `tsconfig.json` has `"noImplicitAny": false` and/or excess-property checks disabled (`"suppressExcessPropertyErrors"` is not a real flag — but `"strict": false` would do it).

Either way, App.tsx `1093: const preCreatedId = (params as any)._tempId as string | undefined` reads it back via cast. Type-unsafe, works at runtime.

**Two cleaner options:**

- **A. Add `_tempId?: string` to `GenerationParams`** in api.ts and types.ts. Minimal change, but pollutes the API contract type with a UI-only field, and the backend would need to know to ignore it (or the App.tsx whitelist could simply drop it before calling `generateApi.startGeneration` — which it already does, since the whitelist doesn't include `_tempId`).
- **B. Pass `tempId` as a separate arg.** Change the callback signature: `onGenerate: (params: GenerationParams, tempId: string) => void`. Keeps type-safety and concerns separated. Recommended.

Refs: `app/components/CreatePanel.tsx:34, 1617, 1731`, `app/App.tsx:993, 117-135`.

Diff sketch (option B):

```ts
// CreatePanel.tsx:34
- onGenerate: (params: GenerationParams) => void;
+ onGenerate: (params: GenerationParams, tempId: string) => void;

// 1616 — drop _tempId from literal, pass second arg
- onGenerate(effectiveCustomMode ? { _tempId: tempIdForThisJob, ... } : { _tempId: ..., ... });
+ onGenerate(effectiveCustomMode ? { ... } : { ... }, tempIdForThisJob);

// App.tsx:993 — read second arg, drop the cast
- const preCreatedId = (params as any)._tempId as string | undefined;
+ // accept tempId as the second arg of handleGenerate
```

## 16. `Song` type duplication: `app/types.ts:Song` vs `api.ts:Song` · **MIN**

`app/types.ts:1-32` declares one `Song`. `app/services/api.ts:89-117` declares another `Song`, used by `transformSongs` and exported by `songsApi.*` calls. They have different shapes:

- `types.ts:Song.duration: string` (formatted "M:SS"); `api.ts:Song.duration?: number` (raw seconds).
- `types.ts:Song.coverUrl: string` (required); `api.ts:Song.cover_url?: string` (optional snake-case).
- `types.ts:Song.tags: string[]` (required); `api.ts:Song.tags: string[]` (required) — same.
- `types.ts:Song.createdAt: Date`; `api.ts:Song.created_at: string`.
- `types.ts:Song.bpm?: number`; `api.ts:Song.bpm?: number` — same.
- `api.ts:Song.openrouterModel?: string` (no `null`); `types.ts:Song.openrouterModel?: string | null`.

The transformer at `api.ts:120-136` doesn't actually convert into `types.ts:Song` shape — it returns `api.ts:Song` (camelCase aliases bolted on). The conversion to UI-facing `Song` happens implicitly elsewhere via `s.duration && s.duration > 0 ? \`${...}:${...}\` : '0:00'` patterns sprinkled through the codebase (e.g. `api.ts:211` in `updateSong`).

**Fix:** rename `api.ts:Song` to `SongRow` or `SongApiResponse` to make the layering explicit. Or fully separate: `SongDb` (snake), `SongUi` (camel + Date + formatted strings), with a single transform fn.

Refs: `app/types.ts:1-32`, `app/services/api.ts:89-117,120-240`.
