**However: backend still ignores DCW + FlowEdit + retake + loraLoaded.** `app/server/src/routes/generate.ts:301-374` destructures `req.body as GenerateBody` with NO entries for `dcwEnabled`, `dcwMode`, `dcwScaler`, `dcwHighScaler`, `dcwWavelet`, `retakeSeed`, `retakeVariance`, `flowEditMorph`, `flowEditSourceCaption`, `flowEditSourceLyrics`, `flowEditNMin`, `flowEditNMax`, `flowEditNAvg`, `loraLoaded`. So the wire shape is correct now; the user-controllable knobs still don't influence audio generation. This downgrades the original "user changes retake variance and nothing happens" symptom from "silently dropped at frontend layer" to "silently dropped at backend layer" — net effect identical for users until backend handlers are wired.

Verdict: **VERIFIED-FIXED on App.tsx; backend wire-through still DEFERRED.** Recommend follow-up: either wire DCW/FlowEdit/retake into `GenerateBody` destructure + pass to `acestep.ts` invocation, or remove the UI controls.

## R2.§12 — New regressions hunted

### R2.§12a — `(params as any)` casts compound the type lie · **NEW-MIN**

The 15 new whitelist additions and `openrouterModel`/`pollinations`/`_tempId` all use `(params as any).field`. The full payload object literal is also closed with `} as any, token)` (`App.tsx:1170`). This means the `startGeneration` argument is now untyped on entry — TS can no longer catch a typo on either side of the call. Functionally fine; type-system blind-spot widened.

Mitigation: extending `app/services/api.ts:GenerationParams` with the 14 dropped fields + `_tempId?: string` would let the inner casts and the trailing `as any` go away. Since CreatePanel imports `GenerationParams` from `app/types.ts`, that type would also need the same fields, OR CreatePanel's import switched to `from '../services/api'`. Either is a small, safe refactor.

### R2.§12b — `mp3Bitrate` / `mp3SampleRate` whitelist forwarding without UI input · **NEW-NIT**

Whitelist forwards `mp3Bitrate`/`mp3SampleRate` (`App.tsx:1133-1134`). Custom-mode CreatePanel builds them (`CreatePanel.tsx:1733-1734`); simple-mode does not. Simple-mode runs send `undefined` for both — backend default kicks in. Cosmetic.

### R2.§12c — Stale `prompt` field forwarding · **NEW-MIN**

`prompt: (params as any).prompt` at `App.tsx:1149` and the same in CreatePanel (`1630, 1745`). Backend `generate.ts:301-374` does NOT destructure `prompt`. So this field is sent on every request, ignored by backend, ends up serialized into `generation_jobs.params` as a duplicate of `lyrics` (custom mode) or `songDescription` (simple mode). Wastes a field. Could be dropped.

## Verdict

Round 2 cleanly addresses the **practical user-visible** part of MAJ §2 — the 15 dropped fields now leave the browser. But:

- The MIN §5 (`_tempId`) commit message is **misleading** — the claimed api.ts type-add did NOT happen. Cast still in place.
- The other 7 deferred findings are reasonable to defer for a local app.
- 3 new minor footguns introduced/highlighted (more `as any` casts, dead `prompt` field, mp3 forwarding asymmetry).
- Backend is still the bottleneck for DCW/FlowEdit/retake/loraLoaded UI controls — they reach the server but the server doesn't destructure them. Recommend either wiring or removing those UI controls in the next pass.

Top three for next round:
1. Either wire DCW/FlowEdit/retake/loraLoaded into `GenerateBody` + `acestep.ts`, or remove the orphan UI controls.
2. Actually add `_tempId?: string` + the 14 dropped fields to `app/services/api.ts:GenerationParams` (and switch CreatePanel's import or duplicate in `app/types.ts`) to delete the 18 `(params as any)` casts.
3. Drop redundant `prompt:` forwarding (or have backend read it as the canonical source for user-entered text).
