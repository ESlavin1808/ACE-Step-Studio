## H1 — HIGH — `cancelAllGenerations` does NOT drain queue waiters → permanent pre-flight deadlock

**Refs:** `App.tsx:808-827`, `App.tsx:92-103`, `App.tsx:761` (the only `drainQueueWaiters()` call)

**Trace:**
1. User clicks Создать → click 1 reaches `beginPollingJob` → `activeJobsRef.size === 1`.
2. User clicks Создать AGAIN → CreatePanel's chain step extends, awaits `waitForJobsToDrain()`. Resolver pushed to `queueDrainResolversRef.current`. The promise is parked.
3. User clicks Cancel All. `cancelAllGenerations` clears `activeJobsRef`, sets `activeJobCount=0`, removes temp songs — **but never invokes `drainQueueWaiters()`**.
4. The parked resolver is now orphaned. Click 2's chain step `await waitForJobsToDrain()` hangs forever.
5. `llmPreflightQueueRef.current` is now a permanently-pending promise. **Every subsequent click** appends `.then(...)` to it; none of those `.then` callbacks ever fire.
6. The `incrementPendingClicks(slotsClaimed)` from those subsequent clicks never gets released (the click flow blocks at `await llmPreflightQueueRef.current` line 1566 before reaching `releaseClaimedSlots()`). **Pending counter monotonically grows**; the N/10 badge ratchets up forever; placeholder cards stuck on `stageWaitingInQueue` accumulate at the top of the song list.
7. The only escape is page reload.

`cleanupJob` (line 744-762) correctly drains. `resetSingleJob` (785-806) does NOT drain either (same class of bug, milder because user usually only resets ONE job and the others can still drain it normally). `cancelAllGenerations` is the catastrophic one because by definition it clears the WHOLE map, so no remaining job will ever call `drainQueueWaiters` for us.

**Fix sketch:**
```diff
@@ App.tsx:823 cancelAllGenerations
   activeJobsRef.current.clear();
   setSongs(prev => prev.filter(s => !tempIds.has(s.id)));
   setActiveJobCount(0);
   setIsGenerating(false);
+  // Release any chain steps parked on waitForJobsToDrain — without this,
+  // the next click's pre-flight hangs forever.
+  drainQueueWaiters();
+  // Optional: reset the chain itself in case any step had already resolved
+  // past the drain barrier into client.generate() and is still in flight.
+  // llmPreflightQueueRef.current = Promise.resolve(null);  // see H2
 }, [token]);
```
And add `drainQueueWaiters` to deps. `resetSingleJob` should call `drainQueueWaiters()` after the `delete()` on parity with `cleanupJob`.

## H2 — HIGH — Chain poisoning: any throw outside the inner `try` makes `llmPreflightQueueRef` permanently rejected

**Refs:** `CreatePanel.tsx:1538-1572`

The chain step is:
```ts
llmPreflightQueueRef.current = llmPreflightQueueRef.current.then(async () => {
  if (waitForJobsToDrain) {
    try { await waitForJobsToDrain(); } catch { /* swallowed */ }
  }
  if (updateTempSongForClick) {
    tempIds.forEach(id => updateTempSongForClick(id, { stage: '...' }));   // ← (a)
  }
  try {
    const client = new OpenRouterProvider();                                // ← (b)
    ...
    return await client.generate(...);
  } catch (e) { return null; }
});
```

The inner `try` only wraps `new OpenRouterProvider() + client.generate()`. Two paths bypass it:

- **(a)** `updateTempSongForClick(...)` — if for any reason this React setState throws (it's a `useCallback` over `setSongs(prev => ...)`; if `prev.map` callback ever throws, e.g. a corrupted Song object, the whole chain step rejects).
- **(b)** `new OpenRouterProvider()` constructor throwing (today it's trivial, but a future refactor that reads `localStorage`/`navigator` could throw in an iframe / privacy-mode setting).

Once the chain is rejected, line 1566 `await llmPreflightQueueRef.current` throws → caught by the outer try at `1568` → `releaseClaimedSlots()` + return. **But the ref is still rejected.** Next click:
```ts
llmPreflightQueueRef.current = llmPreflightQueueRef.current.then(async () => {...});
```
`.then()` on a rejected promise without an error handler propagates the rejection. The new chain is also rejected. Pre-flight never recovers across the whole session.

The user perceives: "first error, then NOTHING ever generates again until reload". Outer try catches it cleanly so slots release — but the LLM preflight is dead.

**Fix sketch:**
```diff
- llmPreflightQueueRef.current = llmPreflightQueueRef.current.then(async () => {
+ llmPreflightQueueRef.current = llmPreflightQueueRef.current
+   .catch(() => null)  // poison-pill firewall: previous step's failure must NOT taint this one
+   .then(async () => {
```
Even better, normalize the ref on catch:
```diff
} catch (e) {
   console.error('[Simple+OR] queued pre-flight failed:', e);
+  llmPreflightQueueRef.current = Promise.resolve(null);  // unstick the chain
   releaseClaimedSlots();
   return;
}
```

## M1 — MED — Slot accounting: success-path decrement assumes EXACTLY ONE App.handleGenerate call per slot

**Refs:** `CreatePanel.tsx:1503-1521`, `App.tsx:1145, 1152`

`incrementPendingClicks(slotsClaimed)` (= bulkCount). For-loop fires `bulkCount` `onGenerate(...)` calls. Each App.handleGenerate ends in either:
- success → `decrementPendingClicks(1)` after `beginPollingJob`
- catch → `decrementPendingClicks(1)` then `setSongs(filter)`

So bulkCount=10 expects 10 decrements. Holes:

- **(a) `onGenerate` is never awaited.** `for (let i...)` calls `onGenerate(payload)` synchronously without `await`. App.handleGenerate is `async` — it returns a Promise the loop ignores. If App.handleGenerate has an UNHANDLED throw before its outer try (lines 980-985: auth check `return` only if `!isAuthenticated`; line 994 `tempId` calc; lines 995-1021 setSongs / setSelectedSong), no decrement runs. With `preCreatedId` always set in this path, lines 997-1004 do a setState that could in theory throw (same risk as H2(a)). Then the slot leaks.
- **(b) The outer try in App.handleGenerate (1023-1147) is broad. Only `setIsGenerating`/`setCurrentView`/`setMobileShowList` (985-987) and the placeholder setSongs (997-1019) sit OUTSIDE it.** Probability of throw is ~0 today, but a future addition to that pre-try block would silently leak slots.
- **(c) bulkCount in CreatePanel resets to 1 at line 1768-1770 (`if (bulkCount > 1) setBulkCount(1)`), but `slotsClaimed` was captured into a const at line 1503 — the for-loop and reset use independent values. OK.**

**Fix sketch:**
```diff

+  // Move auth/early-return below or guarantee a slot decrement before each early return.
   if (!isAuthenticated || !token) {
+    // Caller may have claimed a slot; release it.
```

## M2 — MED — `waitForJobsToDrain` only checks `activeJobsRef`, ignoring `pendingClickCount`

**Refs:** `App.tsx:92-103`, comment at 87-90 says queue model = "gen N+1 starts only after gen N is done".

Scenario: user clicks fast 3× while no jobs are yet in `activeJobsRef` (all three are still in pre-flight LLM phase, or click 1 is mid-`startGeneration` POST). For click 2 and click 3:
1. `incrementPendingClicks(1)` runs synchronously.
2. Chain step extends. Inside the step, `await waitForJobsToDrain()` checks `activeJobsRef.current.size === 0` → returns `Promise.resolve()` immediately. **No actual wait.**
3. So click 2's LLM fires in parallel with click 1's LLM (both via separate chain steps that resolved their drain barrier instantly).

Wait — actually the chain itself serializes them. Click 2's chain step doesn't START until click 1's chain step resolves (because `.then()`). So click 2's LLM still waits for click 1's LLM to finish. But it does NOT wait for click 1's audio + cover pipeline (the comment claims it does).

This means the comment at lines 87-90 (`audio + cover`) and the comment at CreatePanel 1528-1531 (`previous track's full pipeline (LLM + audio + cover)`) **are wrong** for the case where click 2's pre-flight chain step fires before click 1's `beginPollingJob` registers a job. Window: click 1's pre-flight (5-30s) + the POST + the response, before `beginPollingJob` at App.tsx:1142. During that window `activeJobsRef.size === 0` and click 2 will sail past the drain barrier.

For practical click-cadence (user clicks "Создать" twice within ~1s), this race fires 100% of the time. So the "audio + cover before next LLM" claim in code comments is essentially a fiction — what actually serializes is the LLM-to-LLM chain via the queue ref. Audio overlap is permitted.

**Severity MED:** the user *might* be relying on this "wait for audio+cover" property. Either fix the implementation (track in-flight POSTs too) or correct the comments.

**Fix sketch (option A — fix comment):** change lines 87-90 + 1528-1531 to "previous LLM call must complete before this one starts; audio/cover stages may run in parallel".

**Fix sketch (option B — fix behavior):** add a separate `inFlightStartsRef = useRef(0)` bumped before `startGeneration` POST and decremented after `beginPollingJob`. Have `waitForJobsToDrain` also check `inFlightStartsRef.current === 0`.

## M3 — MED — `claimedSlotsRemaining` decrement on success path is implicit / fragile

**Refs:** `CreatePanel.tsx:1514-1522, 1777-1780` and App.tsx:1145

The CreatePanel-side closure `releaseClaimedSlots` decrements `claimedSlotsRemaining` slots and zeroes the var. But on the success path, neither CreatePanel nor App resets `claimedSlotsRemaining` — it stays at `slotsClaimed` value. App's `decrementPendingClicks(1)` inside `beginPollingJob` is correct. But if the outer-try in CreatePanel ever throws AFTER the for-loop has fired some onGenerate calls (e.g., the `setBulkCount(1)` at 1768 throws — won't, but conceptually), the catch at 1771 calls `releaseClaimedSlots()` which decrements ALL claimed slots including those already in flight inside App. Result: double-decrement.

Today no throw can happen between the for-loop end and the catch (only `setBulkCount`). LOW probability but adversarial: any future code added between `1762` and `1771` that throws will cause a counter-underrun. `decrementPendingClicks` clamps at 0 (`Math.max(0, c - n)`) so the badge won't go negative — but other slot-aware code (none today, but future) would be broken.

**Fix sketch:**
```diff
@@ CreatePanel.tsx:1761
   }
+  // For-loop completed — slots are now owned by App.handleGenerate's
+  // success/failure paths. Disarm the catch's release.
+  claimedSlotsRemaining = 0;

   if (bulkCount > 1) {
     setBulkCount(1);
   }
   } catch (e) {
     console.error('handleGenerate crashed:', e);
     releaseClaimedSlots();   // now a no-op if for-loop completed cleanly
   }
```

## L1 — LOW — Pre-flight has no cancel surface

**Refs:** `CreatePanel.tsx:1548-1559`, `App.tsx:809-827`

User clicks Создать, OR pre-flight starts (5-30s). User panics, hits Cancel All. Cancel-all iterates `activeJobsRef` — empty during pre-flight. The OR call continues, eventually returns a draft, the chain step resolves, line 1566 `await` returns → for-loop fires `bulkCount` POSTs → `bulkCount` audio jobs spawn. User can NOT prevent this.

The `AbortController` `ac` is created locally inside the chain step and never escapes — there's no way to call `ac.abort()` from outside.

Acceptable for v1 (LLM is short, audio gen is what users actually want to cancel). Documenting only.

## L2 — LOW — `llmPreflightQueueRef` chain memory bound

**Refs:** `CreatePanel.tsx:268, 1538-1564`

Each click extends `.then(...)`. Each new step is a freshly-allocated Promise. After each step settles, JS's promise micro-task drains it; old steps become unreachable via the ref (the ref always points to the LATEST tail). GC reclaims them. Memory is bounded. Confirmed safe. LOW (not a bug — answering the question in the brief).

## L3 — LOW — `tempIds[i]` lifecycle on partial bulk failure is correct

**Refs:** `CreatePanel.tsx:1506-1513` (creation), `App.tsx:993-1004` (promote), `App.tsx:1149` (catch removes by tempId)

For each iteration, `_tempId: tempIds[i]` flows into App.handleGenerate. `preCreatedId = tempId`, the placeholder is promoted (lines 997-1004). On failure, `setSongs(prev => prev.filter(s => s.id !== tempId))` (line 1149) removes that exact placeholder. Other iterations' placeholders are untouched. Good.

One edge case: if BOTH the CreatePanel outer-catch path (`releaseClaimedSlots` → `removeTempSongForClick(id)` for all tempIds) AND App's catch (per-tempId filter) fire for the same tempId, the song is filtered twice harmlessly (`setSongs` filter is idempotent). OK.
