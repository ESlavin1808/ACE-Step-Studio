## R4.§5 — Hunt for new regressions in batch 5

### R4.§5a — `Song.openrouterModel` still drifts · **NEW-NIT** (carry-over emphasis)

Batch 5 aligned `GenerationParams.openrouterModel` to `string | null` everywhere. Did NOT touch the `Song` parallel:

- `app/types.ts:26` — `Song.openrouterModel?: string | null` (UI Song).
- `app/services/api.ts:115` — `Song.openrouterModel?: string` (API Song, no null).

Effect: when `transformSongs` (`api.ts:168, 188, 232`) sets `openrouterModel: s.openrouter_model || s.openrouterModel`, the `||` collapses both `null` and `''` to `undefined`. Downstream UI sees `undefined`, never `null`. Functionally fine. Stale type. Same family as R1 §3 (Song side, not Params side).

Verdict: **NEW-NIT — carry-over from R1 §3, unchanged.**

### R4.§5b — `pollinations` strict `enabled: boolean` required while frontend short-circuit may pass `enabled: false` only · **NEW-MIN**

Both `types.ts:179-190` and `api.ts:402-413` declare `pollinations.enabled: boolean` as required, with all other inner fields optional. CreatePanel's short-circuit (`CreatePanel.tsx:1662` per R1 §4) builds `{ enabled: false }` only — that's a valid shape under both types. ✓

But: now that `app/types.ts:GenerationParams` has the `pollinations` block with strict structural typing, ANY caller building a `GenerationParams` literal that includes `pollinations` is forced to provide `enabled`. Searched for partial pollinations literals — none found at present. Future refactors should be aware.

NOT a regression — just a tighter type. Cosmetic note.

Verdict: **NEW-MIN — informational, no immediate action.**
